## 一、QUIC工作组

QUIC已经是一个正式标准（RFC 9000）了，Chrome、Edge、Facebook、Google都在用。现在工作组在做的不是设计QUIC，而是**解决QUIC真正跑遍全世界时遇到的各种实际问题**。

**我的理解**：
- 2015-2021年：设计这辆新车（QUIC标准制定）
- 2021-2026年：让这辆车能在各种路况（企业网络、移动网络、跨国链路）上都能跑，还要跑得稳、跑得快、让修车师傅（运营商）也能看懂哪坏了


## 二、这次会议的三个核心议题

### 议题一：QMux——让QUIC能走老路（TCP）

**讲者**：Kazuho Oku等  
**关键词**：UDP被墙、TCP fallback、适配层

**背景**：
- QUIC基于UDP，但很多企业网络、公共WiFi会**屏蔽UDP流量**
- 结果：明明开发了基于QUIC的应用，用户却连不上

**讨论的方案**：
加一个叫 **QMux** 的适配层。如果发现UDP走不通，就把QUIC的数据拆开，从老路（TCP）寄过去，到了目的地再重新组装成QUIC。

**技术难点**：
- TCP是字节流，没有“包边界”，怎么让接收方知道一个包从哪开始到哪结束？→ 加一个“记录层”（Record Layer），每块数据前写上长度
- TCP有自己的流控，如果读慢了会死锁怎么办？→ 必须一直读，不能停
- 版本怎么协商？→ 还在吵是用统一ALPN还是复合ALPN

**我的理解**：
这属于是理想和现实的妥协。理想是全网QUIC，现实是很多地方UDP被墙。QMux是让QUIC能优雅降级的“逃生舱”。


### 议题二：低ACK频率下怎么算准最小RTT

**讲者**：多人合作，有点分不清  
**关键词**：minRTT、OWD、ACK频率、拥塞控制

**背景**：
- 为了省电、省带宽，QUIC想**减少ACK的频率**（一个ACK确认多个包）
- 但问题来了：minRTT（最小往返时间）是拥塞控制的关键参数，少了ACK样本，minRTT算不准
- 实验数据：Wi-Fi环境下，minRTT估计偏大 **8%-18%**

**讨论的方案**：
不用RTT样本，改用**单向延迟（OWD）**来校准：
- 发送方在包里打时间戳
- 接收方算每个包的OWD（用自己的本地时钟，只比大小，不要同步）
- 找出OWD最小的那个包，把它的时间戳告诉发送方
- 发送方就能算出真正的minRTT

**需要改进**：
- 加个传输参数：双方协商是否支持
- TIMESTAMP帧：传出发时间戳（复用已有草案）
- MINOWD-ACK帧：新帧，报告最小OWD的包的信息

**我的理解**：
这是“用代理变量解决问题”的典型例子——直接测不准，就找个相关的东西间接测。这种思路在AI里也很常见（比如用logits估置信度）。


### 议题三：显式测量——给运营商开个小窗户

**讲者**：Alexandre Ferrieux等  
**关键词**：可观测性、隐私、测量、EFMP

**背景**：
- QUIC把所有东西都加密了，运营商**啥也看不见**
- 网络出问题了，运营商没法排查（不知道是用户网不好还是自己设备坏了）
- 但用户隐私又不能不要

**讨论的方案**：
专门设计一种**测量包（EFMP）**，走在正常QUIC包前面，包里留几个bit给测量用（比如Q bit测丢包，L bit测拥塞事件）。

**EFMP包结构**（简化版）：
```
EFMP Packet {
  Q Bit (1),      // sQuare bit，测丢包率
  L Bit (1),      // Loss event bit，测拥塞事件
  Spin Bit (1),   // 测RTT
  Connection ID,  // 和正常包一样，保证路由正确
  ...
}
```

**问题和权衡**：
| 问题 | 说明 |
|------|------|
| **开销** | 每个包多8字节以上 |
| **隐私** | 开了测量的流量容易被识别、被跟踪 |
| **伪造** | 端点可以不按算法设bit，所以不能基于测量做自动策略 |
| **篡改** | 测量包不加密，但端点可以检测到篡改 |

**我的理解**：
这是隐私和可维护性之间的拉锯战。QUIC太保护隐私，把运营商逼急了；现在要往回退一步，给个小窗户，但又不能开太大。这种平衡很难拿捏。


## 三、这三个议题的关系

```
QUIC要跑遍全世界
    ├─ 网络环境太复杂，UDP常被墙 → QMux（让QUIC能走TCP）
    ├─ 性能要优化，但优化有副作用 → minRTT校准（解决副作用）
    └─ 运营商看不见，没法排障 → 显式测量（给个小窗户）
```

- **QMux**：解决“能不能连上”的问题
- **minRTT校准**：解决“连上后跑得稳不稳”的问题
- **显式测量**：解决“跑出问题后能不能查”的问题


## 四、和我的方向（AI+Network）的连接

1. **QMux的“适配层”思维**：AI模型也要适配不同硬件（GPU/TPU/NPU），这种“加一层抽象”的思路可以迁移。

2. **minRTT校准的“代理测量”**：AI里也有很多不能直接测的量（如模型置信度），需要用可测的量去估计。思路一样。

3. **显式测量的“可观测性”**：AI服务也要监控，但现在的监控大多是业务层（QPS、延迟）。如果能像QUIC这样在传输层加测量，可能会发现一些业务层看不到的问题。

4. **反馈信息的设计**：minRTT校准里讨论“反馈时间戳还是反馈包号”，本质是“信息在哪层处理”的问题。AI系统里也有类似问题：梯度在端上算还是中心算？推理在端上做还是云上做？


## 五、我的收获

> “QUIC现在已经是个正式标准了，工作组在解决它真正落地时遇到的问题。一个是QMux，让QUIC在UDP被墙时能降级走TCP，保证连通性。另一个是低ACK频率下的minRTT校准问题——减少ACK会bias minRTT估计，他们用单向延迟来找真正最小的RTT样本。还有个是显式测量，给运营商开个小窗户看网络状态，但要平衡隐私和可观测性。这些讨论让我看到协议设计不是一锤子买卖，标准发布后还有很多填坑的工作，而且每个方案都有权衡。”


## 六、术语表（个人理解版）

| 术语 | 个人理解 |
|------|---------|
| **QMux** | 让QUIC能走TCP的适配层，解决UDP被墙的问题 |
| **Record Layer** | 在TCP字节流上划出“包边界”，告诉接收方一块数据从哪到哪 |
| **minRTT** | 最小往返时间，拥塞控制的关键参数 |
| **OWD** | 单向延迟，用来间接算RTT的代理变量 |
| **ACK频率** | 多久发一次确认回执，省带宽但会影响RTT测算 |
| **EFMP** | 显式测量包，专门用来给运营商看网络状态 |
| **Spin Bit** | 测RTT的标记位 |
| **sQuare bit** | 测丢包率的标记位 |
| **Loss event bit** | 测拥塞事件的标记位 |


## 七、总结

QUIC工作组现在不是在画大饼，而是在解决**真实世界里的真实问题**：
- 网络环境不理想（UDP被墙）→ QMux
- 优化有副作用（minRTT不准）→ 新算法校准
- 隐私和可维护性冲突（运营商看不见）→ 显式测量

这些问题，不到真跑起来的那一刻，是想不到的。这也是为什么QUIC从标准发布（2021年）到现在5年了，工作组还在开会讨论。
