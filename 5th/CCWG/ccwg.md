## 一、CCWG简介

**CCWG（拥塞控制工作组）** 是IETF里专门研究**拥塞控制算法**的团队。

**拥塞控制**
- 就是让发送数据的人**知道路况怎么样，该开多快，该不该减速**
- 好比开车：路况好就加速，堵车就减速，不能一直地板油

**为什么重要**？
- 没有拥塞控制，网络早就瘫了（大家都猛发，互相挤爆）
- 但不同的网络（WiFi、5G、卫星、光纤）需要不同的开车策略

**CCWG的任务**：讨论、测试、标准化各种拥塞控制算法，让它们能在真实世界里好好工作。

## 三、议题一：SCReAM v2——给视频用的拥塞控制

**文件**：`slides-125-ccwg-scream-v2-01.pdf`  
**讲者**：Ingemar Johansson（爱立信）

### 这会在讲什么

SCReAM 是一个专门为**实时视频/音频**设计的拥塞控制算法。v2 版本在原有基础上做了优化。

### 解决什么问题

**视频传输的痛点**：
- 视频编码器反应慢（想降码率要等半天）
- 关键帧（I-frame）特别大，容易造成突发
- 既要低延迟，又要高画质，很难平衡

**v2 的新招**：

| 新特性 | 作用 |
|--------|------|
| **qdelay_dev_norm** | 新指标，衡量拥塞程度 |
| **自适应 ref_wnd_overhead** | 拥塞时限制发送，不拥塞时允许更多突发 |
| **限制 ref_wnd 增长** | 拥塞时慢点加速 |
| **时钟漂移补偿** | 解决两端时钟不同步的问题 |

### 进展

- 已经在 **Chrome Canary** 里实现了（开发者版本）
- 可以在 `chrome://webrtc-internals/` 里看日志
- 有个 L4S 测试网站：`https://l4s-testbed-4smqe.ondigitalocean.app/`

### 我的理解

SCReAM 是在解决视频编码器太慢和网络变化太快之间的矛盾。它在中间搭个桥，让两边能对上话。


## 四、议题二：Rapid Start——让网页加载更快

**文件**：`slides-125-ccwg-rapid-start-00.pdf`  
**讲者**：Kazuho Oku（Fastly）

### 这会在讲什么

**问题**：大部分HTTP连接**还没离开慢启动就结束了**。

**慢启动**
- TCP一开始不知道路况，只能慢慢试探，每次翻倍加速（1→2→4→8）
- 问题：很多网页很小，还没加速到最优，就传输完了

**Rapid Start 的核心思想**：别那么慢，直接多翻一点！

### 三个大招

| 招数 | 做法 | 效果 |
|------|------|------|
| **Full-RTT pacing** | 第一个RTT全程发数据，不空闲 | 充分利用带宽 |
| **3倍增长** | 没堵车前，每次翻3倍（不是2倍） | 更快达到最优 |
| **平滑降速** | 堵车时慢慢降，不猛刹 | 避免二次拥堵 |

### 真实数据（Fastly 生产环境测试）

- **TTLB（首字节时间）降低14.7%**
- 不同地区：10.8% ~ 21.5%
- 不同文件大小：10.6% ~ 14.9%
- 代价：丢包率从1.52%升到1.98%（可接受）

### 我的理解

Rapid Start 是在说：**传统慢启动太保守了，现代网络可以更激进一点**。而且他们有真实数据证明有效，不是纸上谈兵。


## 五、议题三：C4——Christian的拥塞控制算法

**文件**：`slides-125-ccwg-presenting-c4-christians-cc-code-02.pdf`  
**讲者**：Christian Huitema（个人开发者）

### Content

C4 是 Christian 自己设计的拥塞控制算法，目标是：
- 对视频友好
- 支持应用限流场景
- 稳定吞吐量
- 低延迟
- 适配WiFi等复杂网络
- **保持简单**

### 核心设计

**用 Max RTT 而不是 min RTT**：
- 大多数算法用 min RTT 做基准
- C4 用 **Max RTT**，因为觉得 max RTT 更能反映真实情况

**四阶段循环**：
```
Recovery（恢复）→ Cruising（巡航）→ Pushing（推进）→ 重复
```
- 巡航时正常发
- 推进时多发一点，试探网络容量

**级联探测**：
- 分成4个探测级别，逐级试探
- 成功了升一级，失败了降一级
- 避免一次探太多把网络搞崩

### 示例

有好几张图展示C4在不同场景下的表现：
- 多媒体连接：稳
- 短连接（用ECN）：稳
- 带宽变化：能自适应
- 和BBR共存：能公平竞争

### 我的理解

这是**个人开发者挑战大厂算法**的典型案例。Christian 是IETF老将，自己写代码、自己测试、自己提草案。一人军团，牛啊！！！

## 六、议题四：Long-haul CNP——长距离网络的拥塞通知

**文件**：`slides-125-ccwg-draft-tian-ccwg-long-haul-cnp-00-00.pdf`  
**讲者**：Tian（没写具体单位，看内容是数据中心方向的）

### Content

**背景**：数据中心分两种
- **DCN**（数据中心网络）：短距离，低延迟
- **DCI**（数据中心互联）：长距离，几百公里到几千公里，RTT > 10ms

**问题**：现有的拥塞通知机制（CNP）是为DCN设计的，在DCI里**反馈太慢**（>20ms），导致缓冲区溢出、吞吐量崩溃。

### 解决方案

**两级触发**：
- **Level 1**：ECN标记（轻度拥塞）
- **Level 2**：Long-haul CNP生成（重度拥塞）

**BDP自适应阈值**：
- 根据链路带宽和RTT动态计算队列阈值
- K_max = R_port × RTT_est × α

**显式控制指令**：
- Notify（通知）、Pause（暂停）、Rate Reduce（降速）、Resume（恢复）

### 包格式

兼容现有标准：
- ICMPv6扩展格式
- RoCEv2向后兼容（用E bit区分标准CNP和长距离CNP）

### 我的理解

这是**把数据中心那套搬到长距离网络时遇到的坑**。短距离好使的方案，长距离就不灵了，得重新设计。


## 七、议题五：BBR Updates——BBR算法的最新进展

**文件**：`slides-125-ccwg-bbr-updates-00.pdf`  
**讲者**：Ian Swett（Google）

### Content

BBR 是 Google 开发的拥塞控制算法，已经在 **2024年10月** 正式成为 CCWG 的工作组草案（draft-ietf-ccwg-bbr）。

### 最新版本（-05）的改动

**逻辑更新**：
- 修复应用限流场景的 pacing 问题
- 处理虚假丢包恢复
- 如果没有SACK，怎么处理丢包
- Drain阶段增加“3 RTT后退出”的规则

**通用化改造**：
- 去掉TCP-specific的变量名
- 适配QUIC等其他传输协议
- 让 BBR 能跨协议使用

### 开放问题

| 问题 | 状态 |
|------|------|
| 要不要加测试用例附录 | 讨论中 |
| 怎么更好地通用化到非TCP | 讨论中 |
| 变量定义太分散 | 讨论中 |

### 我的理解

BBR 已经从一个Google内部算法变成了 **IETF标准候选**。现在的工作是打磨、通用化、让更多人能实现它。


## 八、议题六：LEO卫星网络对拥塞控制的挑战

**文件**：`slides-125-ccwg-analysis-for-the-adverse-effects-of-leo-mobility-on-internet-congestion-control-00.pdf`  
**讲者**：李宗伦（清华大学）

### Content

**LEO卫星网络**（如Starlink）正在改变互联网，但它们给拥塞控制带来了新挑战。

### LEO卫星有啥问题

| 现象 | 原因 | 对拥塞控制的影响 |
|------|------|------------------|
| 容量剧烈变化 | 卫星移动、无线调度 | 传统算法以为是网络变化，其实是路径变了 |
| RTT剧烈变化 | 卫星移动 | 基于RTT的算法（如Vegas）会误判 |
| 非拥塞丢包 | 信号衰减 | 基于丢包的算法（如Cubic）会误降速 |

### 核心洞察

> LEO移动性会改变路径的**基本属性**，但现有拥塞控制算法**不知道路径变了**，还在用老数据判断新情况。

**打个比方**：
- 你从高速路开进市区，路况变了
- 但你的导航还在用高速上的数据算速度
- 结果就是一路误判

### 提出的概念：Path Phase Boundary（PPB）

把路径看成**一系列阶段**，每个阶段有自己的特性。当卫星切换、路径变化时，就是**阶段边界**。

过了边界，老数据就该扔掉，重新学习新阶段的特性。

### 开放问题

- PPB 怎么暴露给传输层？
- 靠网络主动通知？端点自己猜？还是跨层传？
- 这是个需要讨论的问题

### 我的理解

这是**新网络形态（卫星）和老协议设计（拥塞控制）之间的冲突**。设计时没考虑到的情况，现在出现了，得想办法补。


## 九、这些议题是什么关系

```
拥塞控制算法生态
    ├─ 视频场景优化 → SCReAM v2
    ├─ 网页加载优化 → Rapid Start
    ├─ 新算法挑战者 → C4
    ├─ 长距离网络优化 → Long-haul CNP
    ├─ 主流算法演进 → BBR Updates
    └─ 新网络形态冲击 → LEO卫星挑战
```

- **SCReAM、Rapid Start、C4、Long-haul CNP**：各种场景下的新方案
- **BBR Updates**：主流算法在打磨
- **LEO卫星**：新问题出现，提醒大家“你们设计的假设可能过时了”


## 十、总结

CCWG这次会议展示的是**拥塞控制算法的生态现状**：

1. **主流算法在标准化**：BBR正在成为RFC
2. **场景化优化在推进**：视频、网页、长距离各有各的方案
3. **新算法在涌现**：C4这种个人项目也在挑战大厂
4. **新问题在出现**：LEO卫星让所有人重新思考“拥塞”的定义

拥塞控制不是一门“已经解决”的学问，而是在不断适应新网络、新场景的过程中持续演进。
