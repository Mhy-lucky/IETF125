# 🎓 IETF 125 OPSAREA 参会笔记：AI+Network 前沿实践

> **会议**：IETF 125 OPSAREA 分会  
> **时间**：2026年3月  
> **地点**：中国 深圳  

---

## 一、会议背景

IETF（互联网工程任务组）是全球互联网技术标准的制定机构。本次 OPSAREA 分会聚焦 AI 时代网络运营的实践与挑战，中国三大运营商分享了他们在 AI+Network 领域的最新进展、技术瓶颈及对标准化的需求。

---

## 二、中国电信：智能体规模化与管理挑战

**演讲主题**：*OPS Practices and Challenges in the AI Era*

### 2.1 技术进展

| Slide  | core content |
|-------------|-------------|
| *CNOS Core functions* | 云网控制器、数字孪生、网络大模型、能力工厂 |
| *Resource and data sharing* | 实时数据共享，792个API用于资源激活 |
| *Agents* | 能感知、分析、决策的“能力微服务” |
| *Digital employees* | 调度智能体完成特定岗位职责 |

**落地成果：**
- **1000+ 智能体**部署在生产网络中
- **数字员工平台（DEP）**：智能体全生命周期管理
- **智能体通信系统（ACS）**：智能体寻址、命名与协作
- **网络领域大模型**：面向云网运维的专用大模型
- **小启App**：5万员工日常使用的AI统一入口

### 2.2 技术瓶颈

**challenges：**
- **智能体治理缺失**：无统一命名寻址、职责边界模糊
- **安全风险**：数据泄露、工具滥用、提示词注入、合规审计困难
- **多智能体通信**：性能、可靠性、可追溯性无法保证
- **人机权限模糊**：AI权限无明确边界

### 2.3 标准化需求

| 方向 | 需求内容 |
|-----------|-------------|
| **Agent通信协议** | 定义智能体间任务、知识、消息共享机制 |
| **Agent可观测性** | 日志、追踪、指标、多智能体评估体系 |
| **人机权限框架** | 智能体权限的标准化定义与边界控制 |

### 2.4 Keyword
**智能体通信协议 Agent Protocol**

## 三、中国移动：数据爆炸与通信范式重构

**演讲主题**：*Challenges and practices of AIOPS for carrier IP network*

### 3.1 技术进展

| Slide  | Core content |
|-------------|-------------|
| *Drivers for AIOps* | 质量提升、成本降低、基础设施复杂度、AI进展 |
| *Architecture of AIOps* | 从数据到信息再到行动的转化过程 |
| *Example 1: Fault O&M of IP Network* | 60%+故障由AI运维系统几分钟内处理 |
| *Example 2: SRv6 Path Planning* | 每秒1000条SRv6路径计算 |

**落地成果：**
- **智能运维系统**：60%+故障由AI自动处理，多数几分钟内完成
- **私有模型训练**：针对IP网络故障的专用模型
- **SRv6智能规划**：每秒计算1000条最优SRv6路径
- **人机协同机制**：AI决策需人工确认后执行
- **数据蒸馏**：从海量数据中提取高价值信息

### 3.2 技术瓶颈

| Slide  | Content |
|-------------|----------------------|
| *The Data Quality & Volume* | 数据碎片化、格式不一致、数据量过大 |
| *Lack of efficient communication means* | 多厂商问题、通信效率低 |
| *The "Black Box" & AI Hallucination* | 缺乏可解释性、担心自动操作风险 |
| *Some Open Questions* | 带宽/算力开销、语义通信 |

**核心问题：**
- **数据质量危机**：数据碎片化、格式不一致、PB级处理成本
- **多厂商壁垒**：不同厂商设备数据无法标准化
- **通信效率低下**：传“数据”而非“信息”，需多次迭代定位问题
- **AI黑盒问题**：只给结果不解释原因，难以完全信任
- **采集成本失控**：Telemetry数据采集消耗巨大带宽和算力

### 3.3 标准化需求

| 方向 | 需求内容 |
|-----------|-------------|
| **语义通信** | 设备与控制器从“数据传输”转向“语义对话” |
| **边端智能** | 网络设备本地运行小模型，实现分布式智能 |
| **运维网络重构** | 重新定义AI时代的网络管理架构 |
| **语义数据格式** | 为语义通信定义标准化数据格式 |

### 3.4 Keyword
**语义通信 Semantic communication**

## 四、中国联通：网络韧性与全生命周期保障

**演讲主题**：*Intelligent Operation and Maintenance: Practices and Reflections*

### 4.1 技术进展

| Slide  | Core Content |
|-------------|-------------|
| *The Connectivity Gap* | 端到端定位、iFIT随流检测 |
| *Some of Our Practices for Network Resilience* | 数字孪生、流量预测 |
| *Practice 2: IP Network Traffic Monitoring* | 跨域端到端监控 |
| *Intelligent Computing Center O&M Control* | 强化学习+大模型的智能体 |

**落地成果：**
- **数字孪生网络**：数值仿真+模拟仿真融合的双引擎架构
- **割接仿真与回滚**：配置级仿真，支持原子操作和堆栈式回滚
- **智能流量预测**：时空深度神经网络实现端到端流量预测
- **全栈流量监控**：覆盖家庭宽带、移动网、承载网、应用的端到端监控
- **智算中心运维智能体**：基于强化学习+大模型的智能体

### 4.2 技术瓶颈

| Slide  | Content |
|-------------|----------------------|
| *Problem* | 物理-IP-应用三层无法协同仿真 |
| *Problem* | 无法快速生成仿真规则 |
| *Problem* | 宏观预测与微观验证脱节 |
| *The Connectivity Gap* | 多厂商导致无法端到端定位 |
| *The Attribution Gap* | 无法实现实时高维流量分析 |
| *The Service Gap* | 监控与用户体验脱节 |
| *Hidden Risks* | 割接后长尾风险 |
| *Collaboration Efficiency* | 响应和排障成本高 |

**核心问题：**
- **跨层协同难**：物理-IP-应用三层无法协同仿真
- **仿真效率低**：无法快速自动生成多场景仿真规则
- **宏微观割裂**：宏观业务预测与微观设备验证脱节
- **实时性与精度矛盾**：大规模网络无法实现实时+零误差仿真
- **端到端定位难**：多厂商设备故障描述不一致，无法协同定位
- **应用感知缺失**：监控以IP为中心，无法识别具体应用
- **长尾风险**：配置错误可能在数月后引发故障
- **协同成本高**：故障排查依赖人工跨部门协调

### 4.3 标准化需求

| 方向 | 需求内容 |
|-----------|-------------|
| **网络韧性框架** | 定义网络全生命周期韧性能力标准 |
| **深度感知与预测** | 从流量可视化到智能风险预测的演进路径 |
| **闭环仿真验证** | 操作前推演的标准流程和接口 |
| **自适应控制** | 弹性控制的标准定义 |
| **跨厂商故障描述标准** | 统一多厂商设备的故障描述语言 |

### 4.4 Keyword
**全生命周期韧性 Life Cycle Resilience**
**仿真 Digital Twin Interface**

## Tips:中国移动留下的三个开放性问题

**Slide 标题**：*Some Open Questions*

### Question 1
**Should we consider defining entire O&M network framework in AI Era?**  
我们是否应该考虑重新定义 AI 时代的运维网络整体架构？

### Question 2
**Should we consider a new "semantic communication" protocol between devices and controllers — shifting from "data transfer" to intelligent "dialogue"?**  
我们是否应该考虑一种新的“语义通信”协议，让设备和控制器之间从“数据传输”转向智能“对话”？

### Question 3
**Should we define the data format for the aforementioned semantic communication?**  
我们是否应该为上述语义通信定义数据格式？

-这三个问题是中国移动在演讲最后提出的核心思考，指向未来 AI 时代网络管理可能需要的根本性变革，需继续关注。
